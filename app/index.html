You.com Web Search API for LLMs — Live Demo

Give any LLM real-time web search. See the results. Compare the cost.

What This Is

This tool demonstrates how to ground LLMs with real-time web search using You.com's Search API. Two modes:

  • LLM + You.com Playground — run a single query through any LLM + You.com Search and inspect every step: queries issued, results fetched, tokens used, cost breakdown.
  • You.com vs. Native Search — same query, same LLM, two backends: You.com Search API vs. the provider's built-in native search, then a blind cross-model judge decides.

Two Paradigms for LLM + Web Search

There are two fundamentally different ways to combine an LLM with web search. Understanding the distinction matters for evaluating cost, quality, and transparency.

Paradigm 1 — LLM-Directed Search (what this playground uses)

The LLM is in the driver's seat. It decides what to search for, reads the results, decides whether to search again, and synthesizes when it has enough. Retrieval quality is a function of the model's reasoning — a smarter model issues better queries and knows when to dig deeper. The search loop can run multiple rounds.

Paradigm 2 — Search-Then-Synthesize (OpenRouter + Exa, Perplexity, You.com Research API)

The search provider runs retrieval autonomously — it issues queries, fetches pages, ranks results, and assembles a context document. The LLM receives that context as a single input and synthesizes an answer. The LLM never sees raw search results or decides what to fetch. Retrieval quality depends entirely on the search provider's pipeline.

Key limitation: only one query cycle. There is no iterative loop — if the first retrieval misses something, the LLM cannot course-correct by issuing a follow-up search.

How It Works — You.com vs. Native Search

Both paths use Paradigm 1 — LLM-Directed Search: the same model, the same query, the same multi-round tool-use loop. What changes is the search backend. Path A calls the You.com Search API directly; Path B hands the same tool call to the provider's own built-in native web search.

Running both paths identically — same LLM, same reasoning, different retrieval — makes the search backend the only variable. A blind cross-model judge (A/B position randomized) then scores both answers independently, so neither path gets the benefit of appearing first.

Design principle: CLI modules are standalone and work without the UI. The server is a thin HTTP wrapper that imports from run.py and compare.py.

Adding a new model from an existing provider (e.g. a new Claude or GPT version)
→ one entry in comparison/pricing.json + one entry in grounding/run.py GROUNDING_MODELS. compare.py picks it up automatically.

Adding a model via OpenRouter
→ one entry in pricing.json with "provider": "openrouter" and the OpenRouter model slug. No new agent file needed — openrouter_agent.py handles any model.
Set "native_search_tool": {"type": "exa_online"} to enable the Exa native search path (OpenRouter's :online suffix). All current OpenRouter models have this enabled.

Adding a brand-new provider
→ new agent file in grounding/agents/ inheriting the right base class from base_agent.py
→ new branch in run_youdotcom() and run_native() in compare.py
→ entries in pricing.json and GROUNDING_MODELS. No loop logic in compare.py — it stays in base_agent.py.

Model defaults are driven by pricing.json — set "agent_default": true on one entry per provider to control which model is used when an agent is instantiated directly (benchmarks, scripts). The app always passes model explicitly from pricing.json.

Glossary

Total cost
LLM inference cost plus web search cost for the full query. Inference is calculated from input/output token counts and the model's per-token pricing. Search cost is per-call: You.com charges per API call, native search charges per search invocation.
LLM inference
The portion of cost from tokens processed by the LLM (input tokens read + output tokens generated), excluding search fees.
Search
The portion of cost from web search calls. You.com: $5 per 1,000 queries. Native (Anthropic built-in): $10 per 1,000 searches.
Latency (end-to-end)
Wall-clock time from when the query was sent to when the final answer was received, including all search round-trips and LLM generation time.
Web Search API calls
The number of individual web searches executed. For You.com, each search is a separate REST call to the You.com API. For native, each search happens inside the LLM provider's single API call.
LLM round-trips
The number of API calls made to the LLM. For You.com, the LLM is called multiple times in a loop: it requests a search, we execute it, feed results back, and the LLM decides whether to search again or answer. Each iteration is one round-trip. For native, this is always 1 because all searching happens inside a single API call on the provider's servers.
Sources returned
The number of unique source URLs included in the response. More sources can mean broader coverage but also more context tokens for the LLM to process.
Total tokens
The sum of all input and output tokens across all LLM round-trips for this query.
Input tokens
Tokens sent to the LLM, including the system prompt, user query, and any search results fed back into the conversation. For native search, the search result content is billed as input tokens.
Search context tokens
The estimated number of input tokens attributable to search results (as opposed to the base prompt). Higher values mean the LLM is processing more web content, which increases inference cost.
Output tokens
Tokens generated by the LLM in its response, including any intermediate reasoning before the final answer.
Completeness
Judge score (1-5). Does the answer fully address all parts of the question? A score of 5 means every aspect of the query was covered.
Relevance
Judge score (1-5). Is the information on-topic and useful for the question asked? Penalizes off-topic filler or tangential information.
Specificity
Judge score (1-5). Does the answer provide concrete details (dates, names, numbers, context) rather than vague or generic claims?
Citation Quality
Judge score (1-5). Are sources cited, numbered, and traceable? Can a reader verify the claims by following the references?
Judge model
A separate LLM (different from the one being tested) that blindly evaluates both answers without knowing which path produced which. Answers are randomly assigned as "A" and "B" to prevent positional bias.
Tool calls
The expandable log showing real-time activity during each path's execution. For You.com, it shows each search query with parameters (count, freshness, livecrawl). For native, it shows the streaming events from the provider's built-in search (web_search, code_execution, text generation).
Multi-Model You.com — run the same query through up to 5 models in parallel, all using the You.com Search API. Set any slot to None to skip it. Compare answers, token usage, cost, and latency side by side.
