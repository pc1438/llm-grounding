You.com Web Search API for LLMs — Live Demo

Give any LLM real-time web search. See the results. Compare the cost.

What This Is

This tool demonstrates how to ground LLMs with real-time web search using You.com's Search API. It has two modes: LLM + You.com Playground (run a single query through any LLM + You.com Search and see every step) and You.com vs. Native Search (prove that You.com Search is cheaper and comparable quality vs. each provider's built-in native web search).

use-cases/ ├── grounding/ # Standalone — these files work without the rest │ ├── run.py # Entry point: GROUNDING_MODELS config, client + agent factory, CLI + generator API │ ├── base_agent.py # Tool-use loops by API shape: AnthropicAgent, OpenAIResponsesAgent, OpenAICompatibleAgent │ ├── search_tool.py # You.com Search: execute_search(), tool schemas for all 3 API shapes, extract_urls() │ ├── agents/ # Per-provider wrappers (client setup, model defaults, key resolution) │ │ ├── claude_agent.py # → AnthropicAgent │ │ ├── openai_agent.py # → OpenAIResponsesAgent or OpenAICompatibleAgent (factory fn) │ │ ├── qwen_agent.py # → OpenAIResponsesAgent or OpenAICompatibleAgent (factory fn) │ │ ├── kimi_agent.py # → OpenAICompatibleAgent (Moonshot /v1/responses is 404) │ │ └── llama_agent.py # → OpenAICompatibleAgent (Together AI or Ollama) │ ├── demo.py # Interactive chat demo │ └── env.txt # API key template ├── comparison/ # You.com vs. native search comparison │ ├── compare.py # Core: run_youdotcom(), run_native(), run_judge(), calculate_costs() │ ├── pricing.json # Single source of truth for all model pricing, API shapes, and cost history │ ├── benchmark_runner.py # Run 50-question benchmarks, output JSON results │ ├── perplexity_runner.py # Perplexity Agent API (SaC) runner — used by /api/sac-compare │ └── ydc_research_runner.py # You.com Research API runner — used by /api/sac-compare ├── app/ # Web UI — thin HTTP layer on top of grounding/ and comparison/ │ ├── server.py # HTTP server, SSE streaming — imports from run.py and compare.py, no loop logic │ └── index.html # 5-tab UI (About, Playground, Comparison, Multi-Model, SaC) └── decisions/ # ADRs: documented experiments and choices (system prompt, search params, API shape)

Design principle: CLI modules are standalone and work without the UI. The server is a thin HTTP wrapper that imports from run.py and compare.py. Adding a new model from an existing provider (e.g. a new Claude or GPT version) = one entry in comparison/pricing.json + one entry in grounding/run.py GROUNDING_MODELS. Set api_shape to match the provider's loop class ("anthropic", "responses", or "chat_completions") and compare.py picks it up automatically. Adding a brand-new provider = new agent file in grounding/agents/ that inherits the right base class from base_agent.py + _create_client() branch in compare.py + entries in pricing.json and GROUNDING_MODELS. No loop logic needed — it lives in base_agent.py.

Glossary

Total cost
LLM inference cost plus web search cost for the full query. Inference is calculated from input/output token counts and the model's per-token pricing. Search cost is per-call: You.com charges per API call, native search charges per search invocation.
LLM inference
The portion of cost from tokens processed by the LLM (input tokens read + output tokens generated), excluding search fees.
Search
The portion of cost from web search calls. You.com: $5 per 1,000 queries. Native (Anthropic built-in): $10 per 1,000 searches.
Latency (end-to-end)
Wall-clock time from when the query was sent to when the final answer was received, including all search round-trips and LLM generation time.
Web Search API calls
The number of individual web searches executed. For You.com, each search is a separate REST call to the You.com API. For native, each search happens inside the LLM provider's single API call.
LLM round-trips
The number of API calls made to the LLM. For You.com, the LLM is called multiple times in a loop: it requests a search, we execute it, feed results back, and the LLM decides whether to search again or answer. Each iteration is one round-trip. For native, this is always 1 because all searching happens inside a single API call on the provider's servers.
Sources returned
The number of unique source URLs included in the response. More sources can mean broader coverage but also more context tokens for the LLM to process.
Total tokens
The sum of all input and output tokens across all LLM round-trips for this query.
Input tokens
Tokens sent to the LLM, including the system prompt, user query, and any search results fed back into the conversation. For native search, the search result content is billed as input tokens.
Search context tokens
The estimated number of input tokens attributable to search results (as opposed to the base prompt). Higher values mean the LLM is processing more web content, which increases inference cost.
Output tokens
Tokens generated by the LLM in its response, including any intermediate reasoning before the final answer.
Completeness
Judge score (1-5). Does the answer fully address all parts of the question? A score of 5 means every aspect of the query was covered.
Relevance
Judge score (1-5). Is the information on-topic and useful for the question asked? Penalizes off-topic filler or tangential information.
Specificity
Judge score (1-5). Does the answer provide concrete details (dates, names, numbers, context) rather than vague or generic claims?
Citation Quality
Judge score (1-5). Are sources cited, numbered, and traceable? Can a reader verify the claims by following the references?
Judge model
A separate LLM (different from the one being tested) that blindly evaluates both answers without knowing which path produced which. Answers are randomly assigned as "A" and "B" to prevent positional bias.
Tool calls
The expandable log showing real-time activity during each path's execution. For You.com, it shows each search query with parameters (count, freshness, livecrawl). For native, it shows the streaming events from the provider's built-in search (web_search, code_execution, text generation).

Notes on this comparison

Both sides are deep-research APIs — one call, full synthesis
Unlike the basic You.com Search API (which returns snippets for an LLM to read), both APIs here perform multi-step research and return a synthesized markdown answer with citations. You.com Research runs its own internal orchestration; Perplexity's Agent API runs Claude (your specified model) with web_search and fetch_url tools inside Perplexity's infrastructure.
BYOLLM — Perplexity's key differentiator
Perplexity's Agent API lets you specify any frontier model as the orchestrating LLM — here anthropic/claude-sonnet-4-6. Claude drives the tool calls (web_search, fetch_url), decides when to search and what to fetch, and synthesizes the final answer. You own the model choice; Perplexity provides the search infrastructure.
You.com Research vs. Perplexity Agent API — what's opaque vs. transparent
You.com Research is fully opaque — one POST, no visibility into how many searches ran or what was fetched. Perplexity Agent API returns structured output items (search_results, fetch_url_results, message) so you can see exactly what Claude searched and fetched, plus a full token + cost breakdown in usage.cost.
Pricing — both are estimated per-call
Neither API exposes per-token billing in this mode. You.com Research API bundles search + synthesis into a single call (estimate shown; verify at you.com/platform/pricing). Perplexity Agent API bundles Claude inference + SaC infrastructure — see usage.cost in the API response for actual cost when available.
Token counts
You.com Research API does not return token usage — those fields show N/A. Perplexity SaC returns token counts from Claude's inference running inside their infrastructure.
Judge model
GPT-5.4 acts as a blind evaluator. Answers are randomly assigned to "A" and "B" positions to prevent positional bias. The judge evaluates structure and quality — it cannot verify factual accuracy.
